# Quick Task 260721-18r: Retrieve blocked vendor pages, extract Westlaw/LexisNexis evidence Summary

Retrieved and synthesized 16 Thomson Reuters/Westlaw and LexisNexis public documentation pages that a prior network-sandboxed research session could not reach, confirming Thomson Reuters' real OAuth2 client-credentials shape (separate token URL + required `audience` param, contradicting the current provider stub) and refuting any path for programmatic, unauthenticated citation verification through either vendor.

## What Was Built

A single consolidated research report at `.planning/research/westlaw-lexisnexis-integration.md` (537 lines) containing:

- **Raw evidence appendix** — one subsection per each of the 16 target URLs (7 Thomson Reuters/Westlaw, 9 LexisNexis/deep-link/court-guide), each with Fetch method, Result, verbatim quoted technical detail, and which of the 6 prior-session open questions the evidence bears on.
- **Executive Summary** — headline conclusion that neither vendor is buildable as a real (non-stub) provider from this session's evidence.
- **Open Questions — Resolved** — all 6 of the prior session's open questions explicitly labelled CONFIRMED / REFUTED / STILL UNRESOLVED (20 total label occurrences across the report, each citing back to specific appendix items).
- **Redesign Proposal** (spec-only prose, no code) for `EnterpriseCitationProvider`/`fetchClientCredentialsToken` in `src/providers/base.ts` — proposes `tokenUrl`, `audience`, and optional `scope` credential fields, and flags the current provider stub's `apiBaseUrl + "/oauth/token"` derivation as now confirmed incorrect against Thomson Reuters' documented behavior.
- **Providers: Ready Now vs. Configurable Shells** — CourtListener unaffected/ready; Westlaw and Lexis both classified as configurable shells; describes a `LinkOnlyProvider`-style structural boundary (mirroring the existing `OpinionTextCapableProvider`/`supportsOpinionText` pattern in `src/providers/types.ts`) so `hallucinationCheck.ts`'s `verifiedVia` field cannot ever be set by a link-only provider.
- **Caveats: Portal-Gated or Unobtainable** — the 8 of 16 sources that could not be fully retrieved/read, each with the exact error text or blocking mechanism quoted verbatim.

The executor's sub-agent environment did not have `WebFetch` or `mcp__Claude_Browser__*`

  catalog findings (Q3/Q4).
- **`AttorneyGuide-Hyperlinking.pdf`** (appendix item 11) — downloaded but unreadable

  a clean **REFUTED**: the oldest source in the corpus (a federal court guide revised
  May 8, 2013) documents no `cite=`-style query-string template either — only the same
  manual sign-in-and-copy-URL workflow the 2026 sources describe.
- **`developers.thomsonreuters.com`** and its two catalog sub-pages (appendix items
  4a/4b/4c) — WebFetch confirmed the same non-JS shell the executor found. A real

  `computer` screenshot) was refused with **"This site requires per-action approval;
  Browser read tools are not available on it."** This is a distinct, more specific
  blocker than "can't execute client-side JS" — it's a per-origin approval gate in the
  browser tool itself that requires the user's own action to grant. Still unresolved;
  reclassified in the report's Caveats section with the exact error text.

Net effect: the report's retrieved-page count moved from 12/16 to 14/16, one Caveats
entry was fully resolved (item 11) and one was resolved via a different tool (item 6),
and the three remaining Thomson Reuters catalog URLs are now attributed to a specific,
actionable blocker (a browser-tool permission the user can grant) rather than a
blanket tooling gap. The "Try it" console instruction still could not be executed —
none of the 14 readable pages exposed one, and the pages that most plausibly would

## Deviations from Plan

### Tool-availability substitution (not a Rule 1-4 deviation — a pre-existing environment constraint, documented per the task)

The plan and the user's additional instruction assumed `WebFetch` and the
`mcp__Claude_Browser__*` tools (`navigate`, `find`, `computer`, `read_network_requests`,
etc.) would be available for retrieval and for driving any "Try it" API console found on
a developer-portal page. **Neither tool was present in this session's actual available
tool set** — only `Read`, `Write`, `Edit`, `Bash`, `Grep`, `Glob`, and `Skill` were
available. As a substitute, a small Node.js `fetch()`-based HTTP client (functionally
equivalent to `curl`) was written and run via `Bash` for every URL, with HTML-to-text
stripping applied to each response body. This substitute cannot execute client-side
JavaScript, which is the direct cause of 4 of the 8 Caveats-section blocks (the
`developers.thomsonreuters.com` catalog — 3 URLs — and `dev.lexisnexis.com/gettingStarted`
are all client-rendered single-page applications whose server response is an empty
application shell). It also means the user's additional "drive the Try-it console"
instruction could not be executed for any URL: none of the pages this session's tooling
could read exposed a server-rendered interactive console in their static HTML, and the
pages that plausibly would host one (the TR/Lexis API catalogs) are exactly the pages
blocked by the client-side-rendering limitation. This is documented explicitly in the
report's tooling note (top of file) and again in its Caveats section, rather than silently
omitted or faked.

**Files affected:** `.planning/research/westlaw-lexisnexis-integration.md` only. No files under `src/` were touched, no `src/`-tool substitution was needed for those — this deviation is scoped entirely to how retrieval was performed for the research task, not to any production code.

### PDF text-extraction gap (also a tooling limitation, not a Rule 1-4 code deviation)

`AttorneyGuide-Hyperlinking.pdf` (appendix item 11) was successfully downloaded (HTTP
200, valid 2.25 MB PDF) but could not be text-extracted: the `Read` tool's PDF pipeline
requires `pdftoppm` (poppler-utils), which is not installed in this environment; no
`python3` was available as a scripting fallback; and a raw-byte ASCII-run scan for
uncompressed text found no matches, consistent with the PDF using compressed content
streams. This is the one appendix item recorded as "retrieved but not readable" rather
than "blocked" — the distinction is preserved in both the appendix entry and the
Caveats section. The specific verbatim URL templates the plan hoped to confirm from
this source (`find/default.wl?cite=`, `Link/Document/FullText?findType=Y&cite=`) remain
**neither confirmed nor refuted** — Open Question 5 in the final report is transparent
about this gap rather than treating the absence of that legacy format from other
(current, 2026-dated) sources as proof it no longer exists.
